## 自适应采集模式 v6.0.0（19:00）
- **需求冲突**：客户端每天要用 + 云端要秒级数据 → 不能简单短连接
- **方案**：自适应模式（client.py v6.0.0 + main.py v3.0.0 + health.py v2.0.0）
  - **默认**：保持长连接 + 订阅，数据持续推送（秒级），写入 SQLite
  - **让出**：检测到 BadTooManySessions → 自动断开释放 Session，每 30 秒尝试重连
  - **恢复**：客户端断开后自动恢复高速采集
- 实测验证：启动后检测到客户端占用 → 3 次失败后 `session_yielded` → 每 30 秒 `yield_recheck`（正常）
- OPC 服务器 Session 残留问题：客户端断开后 Session 不会立即释放，需要等服务器 SessionTimeout

## 推送超时检测 + 心跳写入 v7.0.0（19:18）
- **问题**：v6.0.0 的 `_collect_loop` 只检查 session 对象是否存在，OPC 服务器异常断开后 session 对象还在内存里，缓存时间戳还在更新 → 死值问题无法发现
- **死值问题**：OPC UA 只在值变化时推送，值不变 = 没推送 = 时间序列空白
- **client.py v7.0.0 + health.py v3.0.0 + storage 增加 source 列**：
  - `_SubHandler` 新增 `client_ref`，每次收到推送更新 `_last_push_time`
  - `_collect_loop` 基于推送超时判断连接是否真正存活（30 秒无推送 = 异常 → 断开重连）
  - 首次连接宽限期 15 秒（等 OPC 服务器推送首批数据）
  - 心跳写入：每 10 秒将缓存值写入 SQLite（`source=heartbeat`），保证时间序列连续
  - 心跳在推送超时或 Session 让出时自动停止
  - `is_connected()` 改为检查推送新鲜度，不再只看 session 对象
  - 新增状态：`push_timeout`、`waiting_first_push`
  - 健康检查返回 `push_age_seconds`
  - SQLite 历史表增加 `source` 列（push / heartbeat），兼容旧表自动 ALTER
