## 09:24 - 看板轮询间隔优化

### 背景
用户询问卡片实时值刷新间隔，原为 3 秒。

### 改动历程

| 时间 | 改动 | 说明 |
|------|------|------|
| 09:24 | 3s → 1s | 简单改 interval |
| 09:32 | 1s → 0.5s | 用户要求进一步加速 |
| 09:40 | 分析 0.5s 风险 | 指出 setInterval 可能造成请求重叠（雪崩） |
| 10:04 | 改 setTimeout 递归 | 防重叠：poll 跑完再等 500ms 才启动下一次 |

### 关键发现
- OPC UA Bridge 使用订阅模式，`create_subscription(500)` → 服务器每 500ms 推送一次
- 数据链路延迟：OPC 500ms + 看板 500ms ≈ 最大 1 秒

### 代码改动（dashboard.html）

**旧**：
```javascript
setInterval(pollRealtime, 1000);
```

**新**：
```javascript
// 防重叠轮询：poll 跑完再等 500ms 才启动下一次
let _pollTimer = null;
async function startPollLoop() {
  try { await pollRealtime(); } catch (_) {}
  _pollTimer = setTimeout(startPollLoop, 500);
}
startPollLoop();
```

## 15:52 - 缓存即时更新优化（减少推送到显示延迟）

### 背景
用户质疑"为什么 OPC UA 推送到达后要等 2s 才更新缓存"，提出直接在回调里立即更新 `_value_cache`，让看板近实时刷新。

### 问题分析
当前 `_SubHandler.datachange_notification` 只把数据追加到 `_write_buffer`，缓存更新在 `_flush_buffer()`（最多 2s 后）才执行。看板读的就是 `_value_cache`，造成额外延迟。

### 改动（client.py）

**改前**：
```python
async def datachange_notification(self, node, val, data):
    ...
    self._buffer.append({...})  # 仅追加
    self._client_ref._last_push_time = time.time()
    # _value_cache 未更新，需等 _flush_buffer（≤2s）
```

**改后**：
```python
async def datachange_notification(self, node, val, data):
    ...
    self._buffer.append({...})  # 仍追加（SQLite 批量写入）
    # 立即更新内存缓存，看板 0.5s 轮询即可读到
    self._client_ref._value_cache[node_id] = {
        "value": val, "quality": quality,
        "timestamp": now_iso, "source": "push",
    }
    self._client_ref._last_push_time = time.time()
```

### 效果

| 环节 | 改前 | 改后 |
|------|------|------|
| OPC UA 推送 → 缓存更新 | ≤ 2.5s | **毫秒级（回调内直接写）** |
| OPC UA 推送 → 用户卡片可见 | ≤ 3s | **≤ 1s** |
| SQLite 落库 | 不变（仍批量） | 不变 |

### 安全性
- asyncio 单线程，无竞态条件
- `_last_push_time` 已在回调里直接写，`_value_cache` 同理
- heartbeat 仍不覆盖 push（保持优先级）
- 服务重启后生效（NSSM stop/start）

### 关键教训 #27
**缓存和 SQLite 落库必须解耦**。OPC UA 推送回调（单线程 asyncio）里直接写 `_value_cache` 零风险，SQLite 批量写入不受影响。不能把"展示性能"和"持久化策略"强行绑定。
